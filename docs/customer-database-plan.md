## Текущий этап

Сейчас сайт может локально собрать:

- товары в корзине;
- сумму заказа;
- имя клиента;
- телефон или WhatsApp;
- город, регион, адрес и комментарий;
- ручной ответ "Откуда узнали о нас?";
- промокод или код рекламы;
- согласие на обратную связь и акции;
- UTM-метки из URL;
- referrer браузера;
- дату первого и последнего рекламного касания на устройстве.

Эти данные сохраняются только в браузере клиента и добавляются в WhatsApp-сообщение менеджеру. Это не требует переписывания магазина, потому что текущий заказ через WhatsApp остается основным сценарием.

Источник клиента не определяется по номеру WhatsApp. Номер телефона нельзя автоматически связывать с Instagram-аккаунтом. Источник определяется по UTM/referrer/ручному выбору клиента/промокоду.

## customers

Назначение: основная карточка клиента после появления backend или регистрации.

Основные поля:

- `id`
- `created_at`
- `updated_at`
- `name`
- `phone`
- `whatsapp`
- `email`
- `city`
- `region`
- `address`
- `default_discount_percent`
- `customer_type` (`retail`, `wholesale`, `manager_created`)
- `wholesale_status` (`none`, `pending`, `approved`, `blocked`)

Уже можно получить с текущего сайта:

- имя;
- телефон/WhatsApp;
- город;
- регион;
- адрес.

Появится только после backend/регистрации:

- стабильный `customer.id`;
- история заказов клиента;
- подтверждение телефона;
- роли и оптовый статус;
- индивидуальная скидка.

## customer_consents

Назначение: хранение согласий клиента на обратную связь, акции и обработку данных.

Основные поля:

- `id`
- `customer_id`
- `consent_type`
- `is_granted`
- `granted_at`
- `revoked_at`
- `source`
- `text_version`

Уже можно получить с текущего сайта:

- чекбокс согласия на обратную связь и акции;
- дату отправки заказа как момент согласия, если checkbox включен.

Появится только после backend/регистрации:

- история изменений согласия;
- отзыв согласия;
- привязка к постоянному клиенту;
- версии юридического текста.

## Почему сейчас не нужно переписывать магазин

Текущий сайт уже выполняет главную функцию: показывает каталог, собирает корзину и отправляет заказ менеджеру в WhatsApp. Первый технический слой добавляет только локальное сохранение рекламного источника и клиентских полей. Это не меняет модель данных каталога, не требует сервера и не ломает статический Cloudflare Pages.

## Как подключить backend позже

Вариант Supabase/PostgreSQL:

- создать таблицы по схеме выше;
- добавить серверную отправку заказа после формирования WhatsApp-сообщения;
- хранить клиентов, заказы, согласия и attribution в PostgreSQL;
- добавить регистрацию и подтверждение телефона отдельным этапом.

Вариант Cloudflare D1:

- оставить Cloudflare Pages;
- добавить Cloudflare Worker/API для приема заказов;
- хранить компактные таблицы в D1;
- позже синхронизировать заказы с 1C или CRM.

Оба варианта можно подключать постепенно: сначала дублировать WhatsApp-заказ в базу, потом добавить кабинет клиента, затем админку и аналитику.

## Риски приватности и согласий

- Нельзя связывать номер WhatsApp с Instagram-аккаунтом без явного источника данных.
- Согласие на акции должно быть отдельным и не обязательным для заказа.
- Нужно хранить только необходимые клиентские данные.
- Для рассылок важно фиксировать дату и источник согласия.
- При появлении backend понадобится политика обработки персональных данных и механизм удаления/изменения клиентских данных.
