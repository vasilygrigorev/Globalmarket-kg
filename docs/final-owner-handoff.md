# Global Market KG — итоговая передача для владельца

Дата: 2026-07-03. Ветка: `collab/preview-baseline`.

Этот документ — краткая сводка простым языком: что готово, что нужно сделать
для публикации, и что нельзя трогать без Codex/разработчика. Технические
детали — в `docs/production-readiness.md`.

## Что готово прямо сейчас

- **Витрина работает**: каталог, карточки товаров, корзина, оформление заказа.
- **Заказ всегда доходит до вас через WhatsApp** — это гарантировано кодом
  (не только описано в документации), даже если техническая часть с базой
  данных не работает или ещё не подключена.
- **Опционально заказ также сохраняется в базу данных** (Supabase) — если эта
  часть включена и настроена, вы получаете и сообщение в WhatsApp, и запись
  заказа в системе. Если не настроена — сайт просто работает как раньше,
  через WhatsApp, без ошибок для покупателя.
- **Админка (панель для менеджера)** готова и протестирована: список
  заказов с фильтрами (статус, период, сумма, поиск), карточка заказа со
  всеми деталями, смена статуса и комментарий менеджера, кнопки «Позвонить»,
  «Копировать телефон/адрес/сводку заказа», экспорт в CSV, работает с
  телефона.
- **Скидка за регистрацию**: 3% от розничной цены для клиентов, которые
  оставили имя и телефон.
- **Бесплатная доставка**: при заказе от 10 000 сом.
- Полный набор автоматических проверок (240 тестов) проходит без ошибок,
  никаких паролей/ключей в коде и в системе контроля версий нет.

## Что ещё нужно сделать, чтобы опубликовать сайт

Первый production-деплой уже выполнен: сайт на `globalmarket.kg` открывается,
каталог и товарные страницы отдаются корректно. Осталось включить сохранение
заказов в базу данных. Это уже не задачи для Claude — их делает Codex или вы
лично, потому что нужен доступ к платным сервисам (Cloudflare, Supabase) и
реальные пароли:

1. Добавить в Production Cloudflare недостающий секрет
   `SUPABASE_SERVICE_ROLE_KEY`. `SUPABASE_URL` и `MANAGER_WHATSAPP` уже
   добавлены.
2. Подтвердить, что в Supabase есть учётная запись менеджера с правами
   администратора.
3. Сделать один настоящий тестовый заказ и проверить: сообщение пришло в
   WhatsApp, заказ появился в базе, заказ виден в админке, статус можно
   поменять.
4. Решить, когда и как код попадёт в GitHub (сейчас 66+ коммитов лежат
   только локально, в GitHub их ещё нет — это отдельное решение, не
   технический блокер).

## Что происходит, когда покупатель оформляет заказ

1. Покупатель заполняет форму на сайте и нажимает «Отправить заказ в
   WhatsApp».
2. Открывается WhatsApp с готовым текстом заказа, отправленным на номер
   `+996706771103` — этот номер задан в настройках сайта и его можно
   поменять.
3. Если техническая часть с базой данных включена и работает — заказ
   параллельно сохраняется в системе, и вы видите его в админ-панели.
4. Если база данных недоступна (сбой, не настроена) — покупатель всё равно
   попадает в WhatsApp, заказ не теряется. Просто он не появится в
   админ-панели, пока база не заработает.

## Что сейчас умеет админка

- Смотреть список всех заказов и фильтровать их (по статусу, дате, сумме,
  имени/телефону).
- Открывать заказ и видеть: товары, сумму, адрес, комментарий клиента,
  откуда пришёл клиент (реклама/источник).
- Менять статус заказа (Новый / Связались / Подтверждён / Выполнен /
  Отменён) и оставлять свой комментарий.
- Быстро звонить клиенту, копировать телефон, адрес или готовую сводку
  заказа для пересылки.
- Выгружать список заказов в CSV (для Excel).
- Работать с телефона — не только с компьютера.

Чего пока нет: истории повторных покупок клиента (карточки клиента),
отзывов, чат-бота, CRM. Это осознанно отложено на будущее и не мешает
запуску.

## Что сейчас делает Петя (фото товаров), а что — нет

- Петя присылает фотографии товаров через Telegram-бота.
- Фото попадают на сервер как есть — их всё ещё нужно вручную рассортировать
  и привязать к нужному товару (карточка + фото спереди + фото сзади для
  обычных товаров; только карточка — для парфюма 5 мл).
- Автоматической загрузки «фото из Telegram → сразу на сайт» нет и не
  planned — каждая партия фото проходит проверку перед публикацией.
- Сейчас реальные фото есть только у части товаров: **92 из 460 (20%)**.
  Это ожидаемо и не мешает запуску — витрина корректно показывает и товары
  без фото, просто с них меньше продаж, пока фото не появится.
- Есть автоматическая защита: система не даст случайно опубликовать «сырое»
  фото из Telegram (черновик, скриншот, контрольный лист) как товарную
  фотографию.

## Что нельзя трогать без согласования с Codex/разработчиком

- Настройки Cloudflare и Supabase (базы данных и хостинга).
- Любые пароли, ключи доступа, файл `admin/config.js`.
- Публикацию сайта на боевом адресе (production deploy).
- Отправку кода в GitHub (git push).
- Изменения в базе данных напрямую через SQL.
- Домен, DNS, оплату сервисов.

Если что-то из этого нужно сделать — попросите Codex подготовить точный
список шагов, не делайте это «на глаз».

## Следующие 3 практических шага

1. Дать Codex/разработчику команду настроить боевое окружение (шаги 1–2 из
   раздела выше) — это разовая техническая работа с паролями, которую нельзя
   делать через чат с ИИ без прямого доступа к панелям Cloudflare/Supabase.
2. После настройки — опубликовать сайт и сделать один тестовый заказ, чтобы
   лично убедиться, что всё работает (шаги 3–4 выше).
3. Отдельно решить (не срочно), когда и как весь код перенести в GitHub —
   это вопрос удобства бэкапа/совместной работы, не блокирует запуск сайта.
